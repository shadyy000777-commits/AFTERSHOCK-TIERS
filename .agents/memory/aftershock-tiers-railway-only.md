---
name: Aftershock Tiers bot — Railway-only as of 2026-07-15
description: Discord bot now runs only on Railway (separate service deployed from the Discord-bot repo); Replit's copy of main.py is disabled via an env var. Explains the data-loss bug that caused this and how the disable works.
---

There are **two separate Railway services** for this project: one runs `railway_server.py` (static website file server, deployed from AFTERSHOCK-TIERS per `Procfile`/`railway.json`), and a second, separate Railway service runs the actual Discord bot (`main.py`), deployed from the **Discord-bot** repo (`push_to_github.sh` pushes `main.py`/`config.py`/`requirements.txt` there). Neither of these is visible/inspectable from this Replit workspace.

**Root-caused bug (2026-07-15):** running the same `DISCORD_TOKEN` bot on both Railway and Replit at once didn't just risk dropped interactions — it caused real data loss. `save_data()`/`load_data()` in `main.py` only ever read/write the *local disk* copy of `tiers_data.json` and never pull from GitHub before writing. So whichever instance (Railway or Replit) had the staler local copy would, on its next save, push that stale copy back over GitHub and silently erase tests the other instance had just added successfully (user saw a real Discord success embed, then the test vanished with zero trace in any of the 3 data repos). This happened at least twice with real tester submissions.

**Fix applied:** `main.py`'s `if __name__ == "__main__":` block now checks `os.getenv("DISABLE_DISCORD_BOT") == "1"` and skips starting the bot thread (Flask website preview still runs) when set. `DISABLE_DISCORD_BOT=1` was set as a **Replit-only** shared env var (`.replit` `[userenv.shared]`, never pushed to GitHub) — Railway's separate bot service has no such var, so it keeps running the bot normally. This makes Railway the sole bot instance without touching Railway's deployment.

**How to apply:** don't re-enable the bot thread in this Replit workspace (don't unset `DISABLE_DISCORD_BOT` or remove the gate) — that would reintroduce the dual-run data-loss bug. The Flask site preview here is safe to use anytime for screenshots/dev work; it no longer starts a second bot gateway session. If tester submissions ever go missing again, first check whether `save_data`'s local-disk-only architecture still lacks a pull-before-write step — that's the deeper fix if dual-run ever needs to come back.
